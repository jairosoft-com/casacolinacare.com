# Design Document: Azure Board Sync

**Implementation Status:** ✅ COMPLETED (Updated for GUID Support)

**Deliverable:** `.claude/skills/cc-azure-board-sync/SKILL.md` - A fully functional Claude Skill for synchronizing BRD/PRD markdown files and JSON metadata files to Azure Boards work items, with bidirectional tracking via JSON file updates. Now supports GUID-based identifiers as the standard format.

## Overview

The azure-board-sync feature is a Claude Skill defined in a SKILL.md file that provides Claude with instructions for synchronizing Business Requirements Documents (BRDs) and Product Requirements Documents (PRDs) to Azure Boards work items. This skill requires TWO input files and performs bidirectional synchronization:

**Required Input Files:**
1. **Markdown file** (e.g., `BRD_PRD.md` or `prd.md`) - Contains business objectives, user stories, and acceptance criteria in narrative format
2. **JSON metadata file** (e.g., `prd.json`) - Contains structured user story data with GUID-based identifiers, titles, descriptions, and acceptance criteria

**Key Mapping Principle:** One BRD/PRD document is mapped to ONE Feature work item in Azure Boards. The business objectives from the document are included in the Feature's description field, not created as separate Feature work items.

**ID Format:** The skill uses GUID (Globally Unique Identifier) format for feature IDs and user story IDs as the standard, generated by PRD creation skills (cc-gen-brd-prd, cc-gen-prd, cc-gen-prd-lite, cc-gen-prd-task). Sequential numeric IDs (e.g., "001", "US-001") are still supported for backward compatibility.

**Bidirectional Synchronization:**
After creating work items in Azure Boards, the skill updates the JSON metadata file with Azure Board work item IDs, creating a permanent link between local requirements and Azure Board items. This enables tracking, verification, and future synchronization operations.

### Problem Statement

Development teams using Azure Boards for project management often maintain requirements in markdown files (BRDs/PRDs) that must be manually transcribed into Azure work items. This dual-maintenance approach leads to:

- Time waste: Manual creation of dozens of work items per project
- Inconsistency: Human error in transcription and formatting
- Synchronization drift: Requirements documents and Azure Boards falling out of sync
- Lost traceability: Difficulty maintaining links between business objectives and implementation tasks
- No tracking: No record of which local requirements correspond to which Azure Board items

### Solution Approach

The azure-board-sync skill is a SKILL.md file that instructs Claude to:

1. Read and parse BOTH structured BRD/PRD markdown files AND JSON metadata files to extract business objectives, user stories, and acceptance criteria
2. Merge data from both sources to create a complete requirements model
3. Map the entire BRD/PRD document to ONE Feature work item (with business objectives in the description), user stories to User Story work items, and acceptance criteria to Task work items
4. Use Azure DevOps MCP tools to create work items with Markdown-formatted content
5. Establish parent-child relationships to maintain requirement hierarchy
6. Update the JSON metadata file with Azure Board work item IDs for bidirectional tracking
7. Generate a synchronization report for verification and audit

**GUID Support:** The skill is designed to work with GUID-based identifiers (e.g., "a1b2c3d4-e5f6-7890-abcd-ef1234567890") as the standard format for feature IDs and user story IDs. These GUIDs are automatically generated by PRD creation skills and provide globally unique identification across systems. Sequential numeric IDs (e.g., "001", "US-001") remain supported for backward compatibility with existing documents.

### Key Design Principles

- **Prompt-Based**: The skill is a system prompt (SKILL.md), not executable code
- **Convention over Configuration**: Follow established SKILL.md patterns from existing skills
- **Fail-Safe Operation**: Instruct Claude to continue processing remaining items when individual operations fail
- **Traceability**: Maintain explicit links between requirements and work items
- **User-Friendly**: Clear prompts, progress updates, and comprehensive reporting

## Components and Interfaces

### 0. GUID-Based Identifier Support

**Responsibility**: Support GUID (Globally Unique Identifier) format for feature IDs and user story IDs

**GUID Format Standard:**
The skill uses GUID format as the standard for identifiers:
- Feature ID: 36-character GUID (e.g., "a1b2c3d4-e5f6-7890-abcd-ef1234567890")
- User Story ID: 36-character GUID (e.g., "b2c3d4e5-f6a7-8901-bcde-f12345678901")

**Benefits of GUID Format:**
- Globally unique across all systems and projects
- No collision risk when merging requirements from multiple sources
- Enables distributed requirement generation without coordination
- Provides stable identifiers that don't change when requirements are reordered
- Compatible with Azure Boards tagging system

**Backward Compatibility:**
The skill continues to support sequential numeric IDs for backward compatibility:
- Feature IDs: "001", "002", "003", etc.
- User Story IDs: "US-001", "US-002", "US-003", etc.
- Objective IDs: "OBJ-01", "OBJ-02", "GOAL-01", etc.

**PRD Generation Skill Integration:**
The following PRD generation skills now generate GUID-based identifiers by default:
- cc-gen-brd-prd: Generates complete BRD/PRD with GUID-based feature and user story IDs
- cc-gen-prd: Generates PRD with GUID-based feature and user story IDs
- cc-gen-prd-lite: Generates lightweight PRD with GUID-based IDs
- cc-gen-prd-task: Generates task-oriented PRD with GUID-based IDs

**Implementation in SKILL.md:**
The SKILL.md file should:
1. Document GUID format as the standard in examples
2. Show GUID-based IDs in all JSON structure examples
3. Note that sequential IDs are still supported for backward compatibility
4. Include GUIDs in Azure Boards work item tags for traceability

#### Job Steps Section

Per the [Agent Skills Specification](https://agentskills.io/specification), "The Job" section contains the core instructions. This section should be written as a numbered list of clear, actionable steps.

**Required Structure:**

```markdown
## The Job

Follow these steps to synchronize a BRD/PRD file to Azure Boards:

### Step 1: Validate Prerequisites

Before starting, verify:
1. The user has provided a file path to a BRD/PRD markdown file
2. The file exists and is readable
3. Azure DevOps MCP tools are available (check for `mcp_ado_*` tools)
4. The user has provided Azure DevOps organization and project names
5. Connection to Azure DevOps is working (use `mcp_ado_core_list_projects`)

If any prerequisite fails, halt execution and provide a clear error message with guidance on how to fix the issue.

### Step 2: Parse the BRD/PRD File

Read the markdown file and extract structured requirements:

**Business Objectives** (map to Features):
- Look for sections titled "Business Objectives", "Goals", or "Objectives"
- Extract items with IDs matching `OBJ-\d+` or `GOAL-\d+` pattern
- Capture: ID, title (text after colon), full description, success metrics

**User Stories** (map to User Stories):
- Look for headings matching `### US-\d+:` pattern
- Extract: story ID, title, full description
- Parse "As a [role], I want [feature], so that [benefit]" format
- Extract acceptance criteria (bullet lists under the story)
- Extract "Validates:" field to identify parent objectives

**Acceptance Criteria** (map to Tasks):
- Extract checkbox lists `- [ ]` under each user story
- Capture: criterion ID (AC-XXX-XX format), description text
- Extract Gherkin scenarios from code blocks if present

**Error Handling:**
- If file not found: Halt with error message
- If invalid format: Log warnings, continue with partial data
- If missing required fields: Log validation errors, skip malformed items

### Step 3: Map Requirements to Work Items

Transform parsed requirements into Azure Boards work item specifications:

**Business Objective → Feature:**
- Title: `{objective.id}: {objective.title}` (max 255 chars)
- Description: Full objective description + success metrics (HTML format)
- State: "New"
- Tags: "synced-from-prd;business-objective" (semicolon-separated)
- Type: "Feature"

**User Story → User Story:**
- Title: `{story.id}: {story.title}` (max 255 chars)
- Description: Full "As a... I want... so that..." text (HTML format)
- Acceptance Criteria: Formatted list of all criteria (use acceptanceCriteria field)
- State: "New"
- Tags: "synced-from-prd" (semicolon-separated)
- Type: "User Story"

**Acceptance Criterion → Task:**
- Title: `{criterion.id}: {criterion.text}` (truncate if > 255 chars)
- Description: Full criterion text + Gherkin scenario if present (HTML format)
- State: "To Do"
- Tags: "acceptance-criterion;synced-from-prd" (semicolon-separated)
- Type: "Task"

**Validation Rules:**
- Titles must be ≤ 255 characters (Azure limit)
- Descriptions should be converted to HTML format
- Tags must be semicolon-separated with no spaces
- All required fields must be present before creation

### Step 4: Create Work Items in Azure Boards

Use Azure DevOps MCP tools to create work items:

**For each Feature (Business Objective):**
```
Tool: mcp_ado_wit_create_work_item
Parameters:
  organization: {user-provided org}
  project: {user-provided project}
  type: "Feature"
  title: {formatted title}
  description: {formatted description}
  state: "New"
  tags: "synced-from-prd;business-objective"
```
- Capture the returned work item ID and URL
- Store mapping: {source_id} → {azure_work_item_id}
- If creation fails: Log error with full details, continue with next item

**For each User Story:**
```
Tool: mcp_ado_wit_create_work_item
Parameters:
  organization: {user-provided org}
  project: {user-provided project}
  type: "User Story"
  title: {formatted title}
  description: {formatted description}
  acceptanceCriteria: {formatted criteria list}
  state: "New"
  tags: "synced-from-prd"
```
- Capture the returned work item ID and URL
- Store mapping: {source_id} → {azure_work_item_id}
- If creation fails: Log error, continue with next item

**For each Task (Acceptance Criterion):**
```
Tool: mcp_ado_wit_create_work_item
Parameters:
  organization: {user-provided org}
  project: {user-provided project}
  type: "Task"
  title: {formatted title}
  description: {formatted description}
  state: "To Do"
  tags: "acceptance-criterion;synced-from-prd"
```
- Capture the returned work item ID and URL
- Store mapping: {source_id} → {azure_work_item_id}
- If creation fails: Log error, continue with next item

### Step 5: Establish Parent-Child Relationships

Create hierarchical links between work items:

**Link User Stories to Features:**
- For each created User Story, check its "Validates:" field from the source document
- Look up the parent Feature's Azure work item ID from the mapping
- If parent exists:
  ```
  Tool: mcp_ado_wit_work_items_link
  Parameters:
    organization: {user-provided org}
    project: {user-provided project}
    sourceId: {user_story_work_item_id}
    targetId: {feature_work_item_id}
    linkType: "System.LinkTypes.Hierarchy-Reverse"
    comment: "Linked from BRD/PRD sync"
  ```
- If parent not found: Log warning, skip link creation
- If link creation fails: Log warning, continue (link can be added manually)

**Link Tasks to User Stories:**
- For each created Task, extract the parent User Story ID from the task's source ID
  - Example: AC-001-01 → parent is US-001
- Look up the parent User Story's Azure work item ID from the mapping
- If parent exists:
  ```
  Tool: mcp_ado_wit_work_items_link
  Parameters:
    organization: {user-provided org}
    project: {user-provided project}
    sourceId: {task_work_item_id}
    targetId: {user_story_work_item_id}
    linkType: "System.LinkTypes.Hierarchy-Reverse"
    comment: "Linked from BRD/PRD sync"
  ```
- If parent not found: Log warning, skip link creation
- If link creation fails: Log warning, continue

**Validation:**
- Always verify parent exists before attempting to create link
- Use "System.LinkTypes.Hierarchy-Reverse" for parent links (child → parent)
- Continue processing even if individual links fail

### 2. BRD/PRD Parsing Instructions

**Responsibility**: Instructions for Claude on how to extract structured requirements from BOTH markdown and JSON files

**Dual-Source Parsing Instructions in SKILL.md:**

The SKILL.md file should instruct Claude to:

**Step 1: Validate Both Files Exist**
1. Verify markdown file path is provided and file exists
2. Auto-detect or prompt for JSON metadata file:
   - Look for `prd.json` in the same directory
   - Look for `{markdown_basename}_prd.json` (e.g., `Sample_BRD_PRD.md` → `Sample_prd.json`)
   - If not found, prompt user for JSON file path
3. Verify JSON file exists and is readable
4. If either file is missing, halt with clear error message

**Step 2: Parse Markdown File (Business Objectives)**
1. **Section Identification**: Look for markdown headings to identify sections
   - Business Objectives: `## Business Objectives` or `## Goals`
   - Extract objectives with `OBJ-\d+:` or `GOAL-\d+:` pattern

2. **Objective Extraction**:
   - Extract ID, title, and description
   - Capture success metrics if present
   - Store in objectives map: `{ "OBJ-01": { id, title, description, metrics } }`

**Step 3: Parse JSON File (User Stories and Acceptance Criteria)**
1. **JSON Structure Validation**:
   - Verify JSON is valid and parseable
   - Verify required fields exist: `project`, `featureName`, `featureId`, `userStories[]`
   - If invalid, halt with detailed error message

2. **Extract Project Metadata**:
   - `project`: Project name
   - `featureName`: Feature name (use in tags)
   - `featureId`: Feature ID (GUID) - use in tags for traceability
   - `description`: Feature description

3. **Extract User Stories**:
   - For each item in `userStories[]` array:
     - Extract: `id` (GUID), `title`, `description`, `acceptanceCriteria[]`, `priority`, `technicalSpecSection`
     - Store in stories map: `{ "{guid}": { id, title, description, criteria, priority } }`

4. **Extract Acceptance Criteria**:
   - For each user story, extract `acceptanceCriteria[]` array
   - Each criterion is a string describing the acceptance condition
   - Associate criteria with parent user story

**Step 4: Merge Data from Both Sources**
1. Create a unified requirements model combining:
   - Business objectives from markdown
   - User stories from JSON
   - Acceptance criteria from JSON
   - Project metadata from JSON

2. Validate relationships:
   - Check if user stories reference valid objectives (via markdown "Validates:" field if present)
   - Verify all user stories have at least one acceptance criterion

**Error Handling Instructions:**
- If markdown file not found: Report error and halt
- If JSON file not found: Report error and halt
- If JSON is malformed: Report detailed parsing error and halt
- If required JSON fields missing: Report validation error and halt
- If invalid format in markdown: Log warnings and continue with partial data

    'Microsoft.VSTS.Common.Priority': number;  // 1-4 (1=highest)
    'Microsoft.VSTS.Common.AcceptanceCriteria'?: string;  // User Stories only, HTML format
    'System.CreatedDate': string;          // ISO 8601 timestamp
    'System.CreatedBy': string;            // User display name
    'System.ChangedDate': string;          // ISO 8601 timestamp
    'System.ChangedBy': string;            // User display name
  };
  relations?: WorkItemRelation[];          // Parent-child and other links
  url: string;                             // API URL
  _links: {
    self: { href: string };
    workItemUpdates: { href: string };
    workItemRevisions: { href: string };
    workItemHistory: { href: string };
    html: { href: string };                // Web UI URL
    workItemType: { href: string };
    fields: { href: string };
  };
}

interface WorkItemRelation {
  rel: string;  // Link type, e.g., "System.LinkTypes.Hierarchy-Forward" or "System.LinkTypes.Hierarchy-Reverse"
  url: string;  // URL to related work item
  attributes: {
    isLocked: boolean;
    name: string;  // "Parent", "Child", "Related", etc.
    comment?: string;
  };
}
```

**Field Constraints:**
- `System.Title`: Maximum 255 characters (enforced by Azure)
- `System.Description`: HTML format, recommend < 32KB for performance
- `System.Tags`: Semicolon-separated (e.g., "tag1;tag2;tag3"), no spaces around semicolons
- `System.State`: Must match process template states (Agile: New, Active, Resolved, Closed)
- `System.AreaPath` and `System.IterationPath`: Must exist in project, default to project root

**Link Types for Hierarchy:**
- `System.LinkTypes.Hierarchy-Reverse`: Create parent link (child → parent)
- `System.LinkTypes.Hierarchy-Forward`: Create child link (parent → child)
- For this skill, always use `Hierarchy-Reverse` when linking from child to parent

**MCP Tool Response Format:**
The `mcp_ado_wit_create_work_item` tool returns the full work item object above. Key fields to capture:
- `id`: Azure work item ID (for mapping and linking)
- `_links.html.href`: Web UI URL (for reporting)
- `fields['System.Title']`: Verify title was set correctly
- `fields['System.State']`: Verify state was set correctly

## Error Handling

The azure-board-sync skill implements a multi-layered error handling strategy designed to maximize resilience and provide clear feedback to users.


### Error Categories

#### 1. Fatal Errors (Halt Execution)

These errors prevent the skill from proceeding and require user intervention:

- **File Not Found**: Source BRD/PRD file does not exist
  - Error Message: "Error: File not found at path '{filePath}'. Please verify the file path and try again."
  - Action: Halt execution immediately

- **MCP Tools Unavailable**: Azure DevOps MCP tools are not installed or accessible
  - Error Message: "Error: Azure DevOps MCP tools are not available. Please install the Azure DevOps MCP server and try again."
  - Action: Halt execution immediately

- **Connection Failure**: Cannot connect to Azure DevOps organization/project
  - Error Message: "Error: Failed to connect to Azure DevOps. Organization: '{org}', Project: '{project}'. Please verify credentials and project access."
  - Action: Halt execution immediately

#### 2. Recoverable Errors (Continue Processing)

These errors affect individual items but allow the skill to continue:

- **Work Item Creation Failure**: Individual work item fails to create
  - Error Message: "Warning: Failed to create {workItemType} '{title}'. Error: {errorDetails}. Continuing with remaining items."
  - Action: Log error, add to error list, continue processing

- **Link Creation Failure**: Parent-child relationship fails to establish
  - Error Message: "Warning: Failed to link {childType} '{childId}' to parent {parentType} '{parentId}'. Error: {errorDetails}. Link can be created manually."
  - Action: Log warning, continue processing


- **Parent Not Found**: Child work item references non-existent parent
  - Error Message: "Warning: Parent work item '{parentId}' not found for {childType} '{childId}'. Skipping link creation."
  - Action: Log warning, skip link, continue processing

- **Validation Failure**: Work item specification missing required fields
  - Error Message: "Warning: Validation failed for {workItemType} '{sourceId}'. Missing required field: {fieldName}. Skipping work item creation."
  - Action: Log error, skip item, continue processing

#### 3. Parsing Warnings (Non-Critical)

These issues indicate potential problems in the source document but don't prevent processing:

- **Invalid User Story Format**: User story doesn't follow "As a... I want... so that..." format
  - Error Message: "Warning: User story '{storyId}' does not follow standard format. Story will be created but may need manual review."
  - Action: Log warning, create work item with available data

- **Missing Validates Field**: User story lacks "Validates:" field for parent linking
  - Error Message: "Warning: User story '{storyId}' has no 'Validates:' field. Work item will be created but not linked to a Feature."
  - Action: Log warning, create work item without parent link

## GUID-Based Identifier Support

### Overview

The azure-board-sync skill uses GUID (Globally Unique Identifier) format as the standard for feature IDs and user story IDs. This format is automatically generated by PRD creation skills and provides several advantages over sequential numeric IDs.

### GUID Format Specification

**Standard Format:**
- Feature ID: 36-character UUID in format `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`
- User Story ID: 36-character UUID in same format
- Example Feature ID: `a1b2c3d4-e5f6-7890-abcd-ef1234567890`
- Example User Story ID: `b2c3d4e5-f6a7-8901-bcde-f12345678901`

**UUID Version:**
- Uses UUID v4 (random) format
- Complies with RFC 4122 specification
- Generated using cryptographically secure random number generators

### Benefits of GUID Format

1. **Global Uniqueness**: GUIDs are globally unique across all systems, projects, and time periods
2. **No Collision Risk**: Eliminates risk of ID conflicts when merging requirements from multiple sources
3. **Distributed Generation**: Multiple team members can generate requirements independently without coordination
4. **Stable Identifiers**: IDs remain stable even when requirements are reordered or reorganized
5. **System Integration**: Compatible with Azure Boards tagging system and other enterprise tools
6. **Traceability**: Enables precise tracking across multiple systems and tools

### Backward Compatibility

The skill maintains full backward compatibility with sequential numeric ID formats:

**Supported Legacy Formats:**
- Feature IDs: `"001"`, `"002"`, `"003"`, etc.
- User Story IDs: `"US-001"`, `"US-002"`, `"US-003"`, etc.
- Objective IDs: `"OBJ-01"`, `"OBJ-02"`, `"GOAL-01"`, etc.
- Acceptance Criteria IDs: `"AC-001-01"`, `"AC-001-02"`, etc.

**Migration Path:**
- Existing documents with sequential IDs continue to work without modification
- New documents generated by PRD skills use GUID format by default
- Mixed documents (some GUIDs, some sequential) are supported
- No breaking changes to existing workflows

## Azure DevOps MCP Server Configuration

Per the [Azure DevOps MCP Toolset](https://github.com/microsoft/azure-devops-mcp/blob/main/docs/TOOLSET.md), the Azure DevOps MCP server must be configured before using this skill.

### Installation

The Azure DevOps MCP server can be installed via npm:

```bash
npm install -g @microsoft/azure-devops-mcp
```

Or configured directly in Claude Code's MCP settings.

### Configuration

Add to Claude Code's MCP configuration file (`.claude/mcp.json` or user-level config):

```json
{
  "mcpServers": {
    "azure-devops": {
      "command": "npx",
      "args": ["-y", "@microsoft/azure-devops-mcp"],
      "env": {
        "AZURE_DEVOPS_PAT": "${AZURE_DEVOPS_PAT}",
        "AZURE_DEVOPS_ORG": "${AZURE_DEVOPS_ORG}"
      }
    }
  }
}
```

### Environment Variables

Required environment variables:

- `AZURE_DEVOPS_PAT`: Personal Access Token with permissions:
  - Work Items: Read, Write, Manage
  - Project and Team: Read
  - Code: Read (if syncing from repos)

- `AZURE_DEVOPS_ORG` (optional): Default organization name
  - Can be overridden by user input when invoking the skill

### Obtaining a Personal Access Token

1. Navigate to Azure DevOps: `https://dev.azure.com/{organization}`
2. Click on User Settings (top right) → Personal Access Tokens
3. Click "New Token"
4. Set name: "Claude Code MCP"
5. Set expiration: 90 days (or custom)
6. Select scopes:
   - Work Items: Read, Write, Manage
   - Project and Team: Read
7. Click "Create" and copy the token
8. Store in environment variable or `.env` file
