# Requirements Document: GUID Generation for PRD Skills

## Introduction

This feature adds GUID (Globally Unique Identifier) generation capabilities to the four PRD-generating skills in the `.claude/skills/` directory. Currently, these skills use sequential numbering (001, 002, etc.) or date-based IDs for features and user stories. The goal is to switch to GUIDs for better uniqueness and traceability, especially when working with external systems like Azure Boards.

The affected skills are:
- cc-gen-brd-prd - Generates unified BRD/PRD documents
- cc-gen-prd - Generates standalone PRD documents
- cc-gen-prd-lite - Generates streamlined PRD documents
- cc-gen-tech-spec - Generates Technical Design Specifications from PRDs
- cc-gen-prd-task - Converts PRDs to prd.json format for Ralph

## Glossary

- **GUID**: Globally Unique Identifier - a 128-bit identifier that is unique across space and time
- **PRD**: Product Requirements Document - a document describing feature requirements
- **BRD**: Business Requirements Document - a document describing business objectives
- **User Story**: A requirement expressed from the user's perspective (format: "As a [user], I want [feature] so that [benefit]")
- **Feature ID**: A unique identifier for a feature in a PRD
- **User Story ID**: A unique identifier for a user story within a feature (format: US-XXX)
- **cc-gen-brd-prd**: Skill that generates unified BRD/PRD documents
- **cc-gen-prd**: Skill that generates standalone PRD documents
- **cc-gen-prd-lite**: Skill that generates streamlined PRD documents
- **cc-gen-tech-spec**: Skill that generates Technical Design Specifications from PRDs
- **cc-gen-prd-task**: Skill that converts PRDs to prd.json format for Ralph autonomous agent
- **Azure Boards**: External work item tracking system that requires unique IDs
- **SKILL.md**: The instruction file for each skill containing templates and guidance

## Requirements

### Requirement 1: Generate GUIDs for Feature IDs

**User Story:** As a product manager, I want each feature to have a unique GUID, so that I can track features across different systems without ID conflicts.

#### Acceptance Criteria

1. WHEN a PRD is generated by cc-gen-brd-prd, THE Skill SHALL generate a GUID for the Feature ID field
2. WHEN a PRD is generated by cc-gen-prd, THE Skill SHALL generate a GUID for the Feature ID field
3. WHEN a PRD is generated by cc-gen-prd-lite, THE Skill SHALL generate a GUID for the Feature ID field
4. THE GUID SHALL follow the standard UUID v4 format (8-4-4-4-12 hexadecimal characters)
5. THE GUID SHALL be displayed in the Overview section of the PRD as "Feature ID: [GUID]"

### Requirement 2: Generate User Story IDs Based on Feature GUID

**User Story:** As a developer, I want each user story ID to reference its parent feature GUID, so that I can trace stories back to their feature unambiguously across tools and systems.

#### Acceptance Criteria

1. WHEN a user story is created in a PRD with a Feature GUID, THE User Story ID SHALL use the format "US-[FEATURE-GUID]-01", "US-[FEATURE-GUID]-02", etc.
2. THE sequence number SHALL be zero-padded to 2 digits (01, 02, 03, etc.)
3. THE User Story heading SHALL use the format "### US-[FEATURE-GUID]-01: [Title]"
4. WHEN multiple user stories exist in a PRD, THE System SHALL increment the sequence number for each story
5. THE User Story ID SHALL be preserved when converting to prd.json format
6. WHEN a PRD uses the old sequential format (US-001), THE Skills SHALL continue to support it for backward compatibility

### Requirement 3: Generate Functional Requirement IDs Based on Feature GUID

**User Story:** As a business analyst, I want each functional requirement ID to reference its parent feature GUID, so that I can trace requirements back to their feature unambiguously.

#### Acceptance Criteria

1. WHEN a functional requirement is created in a PRD with a Feature GUID, THE Functional Requirement ID SHALL use the format "FR-[FEATURE-GUID]-01", "FR-[FEATURE-GUID]-02", etc.
2. THE sequence number SHALL be zero-padded to 2 digits (01, 02, 03, etc.)
3. WHEN multiple functional requirements exist in a PRD, THE System SHALL increment the sequence number for each requirement
4. WHEN a PRD uses the old sequential format (FR-1, FR-2), THE Skills SHALL continue to support it for backward compatibility
5. THE Functional Requirement SHALL include "Supports: US-[FEATURE-GUID]-XX" notation to link to user stories

### Requirement 4: Generate Business Objective IDs Based on Feature GUID

**User Story:** As a product manager, I want each business objective ID to reference its parent feature GUID, so that I can trace objectives back to their feature unambiguously.

#### Acceptance Criteria

1. WHEN a business objective is created in a BRD/PRD with a Feature GUID, THE Business Objective ID SHALL use the format "OBJ-[FEATURE-GUID]-01", "OBJ-[FEATURE-GUID]-02", etc.
2. THE sequence number SHALL be zero-padded to 2 digits (01, 02, 03, etc.)
3. WHEN multiple business objectives exist in a document, THE System SHALL increment the sequence number for each objective
4. WHEN a PRD uses the old sequential format (OBJ-01, OBJ-02), THE Skills SHALL continue to support it for backward compatibility
5. THE User Stories SHALL include "Validates: OBJ-[FEATURE-GUID]-XX" notation to link back to business objectives

### Requirement 5: Generate Acceptance Criteria IDs Based on User Story IDs

**User Story:** As a developer, I want acceptance criteria IDs to reference their parent user story ID, so that I can trace each criterion back to its story unambiguously.

#### Acceptance Criteria

1. WHEN a user story uses the format "US-[FEATURE-GUID]-01", THE Acceptance Criteria IDs SHALL use the format "AC-[FEATURE-GUID]-01-01", "AC-[FEATURE-GUID]-01-02", etc.
2. THE increment number SHALL be zero-padded to 2 digits (01, 02, 03, etc.)
3. WHEN a user story uses sequential format (US-001), THE Acceptance Criteria IDs SHALL continue to use "AC-001-01", "AC-001-02" format for backward compatibility
4. THE Skills SHALL automatically derive the AC ID prefix from the user story ID
5. WHEN examples show Acceptance Criteria, THE Examples SHALL demonstrate the GUID-based format

### Requirement 6: Generate Test Case IDs Based on User Story IDs

**User Story:** As a QA engineer, I want test case IDs to reference their parent user story ID, so that I can trace each test back to the story it validates.

#### Acceptance Criteria

1. WHEN a user story uses the format "US-[FEATURE-GUID]-01", THE Test Case IDs SHALL use the format "TC-[FEATURE-GUID]-01-01", "TC-[FEATURE-GUID]-01-02", etc.
2. THE increment number SHALL be zero-padded to 2 digits (01, 02, 03, etc.)
3. WHEN a user story uses sequential format (US-001), THE Test Case IDs SHALL continue to use "TC-001-01", "TC-001-02" format for backward compatibility
4. THE Skills SHALL automatically derive the TC ID prefix from the user story ID
5. WHEN examples show Test Cases, THE Examples SHALL demonstrate the GUID-based format with the "Validates:" field showing the user story and acceptance criteria references

### Requirement 7: Update cc-gen-tech-spec to Handle GUID-based IDs

**User Story:** As a technical architect, I want the Technical Design Specification to reference GUID-based IDs from the PRD, so that the technical specs maintain complete traceability to requirements.

#### Acceptance Criteria

1. WHEN cc-gen-tech-spec reads a PRD with GUID-based user story IDs, THE Skill SHALL preserve those IDs in the TECH_SPEC.md
2. THE Design Details section SHALL use headings in the format "### 3.X US-[FEATURE-GUID]-01: [Title]"
3. WHEN referencing acceptance criteria from the PRD, THE Skill SHALL use the format "AC-[FEATURE-GUID]-01-01"
4. WHEN referencing functional requirements from the PRD, THE Skill SHALL use the format "FR-[FEATURE-GUID]-01"
5. WHEN referencing business objectives from the BRD/PRD, THE Skill SHALL use the format "OBJ-[FEATURE-GUID]-01"
6. THE Testing Strategies section SHALL reference test cases using the format "TC-[FEATURE-GUID]-01-01"
7. THE Markdown heading anchors SHALL be generated correctly for GUID-based IDs (e.g., `#3x-us-featureguid-01-title`)
8. WHEN a PRD uses the old sequential format (US-001, AC-001-01, etc.), THE Skill SHALL continue to support it for backward compatibility
9. THE Technical specifications SHALL maintain the "Validates:" and "Supports:" traceability links using GUID-based IDs

### Requirement 8: Update cc-gen-prd-task to Handle GUID-based IDs

**User Story:** As a developer using Ralph, I want the prd.json converter to handle GUID-based user story IDs, so that I can use the new ID format without breaking the conversion process.

#### Acceptance Criteria

1. WHEN cc-gen-prd-task converts a PRD with GUID-based user story IDs, THE Skill SHALL extract the ID from the heading format "### US-[FEATURE-GUID]-01: [Title]"
2. THE Skill SHALL populate the "id" field in prd.json with the format "US-[FEATURE-GUID]-01"
3. WHEN a PRD uses the old sequential format (US-001, US-002), THE Skill SHALL continue to support it for backward compatibility
4. THE Skill SHALL detect the ID format automatically based on the heading pattern
5. THE featureId field in prd.json SHALL accept GUID format in addition to sequential numbers

### Requirement 9: Update Skill Templates and Examples

**User Story:** As a skill user, I want the templates and examples to show GUID usage, so that I understand how to work with the new ID format.

#### Acceptance Criteria

1. THE cc-gen-brd-prd SKILL.md SHALL include examples showing GUID format for Feature ID, Business Objective IDs (OBJ-[FEATURE-GUID]-01), User Story IDs (US-[FEATURE-GUID]-01), Functional Requirement IDs (FR-[FEATURE-GUID]-01), Acceptance Criteria IDs (AC-[FEATURE-GUID]-01-01), and Test Case IDs (TC-[FEATURE-GUID]-01-01)
2. THE cc-gen-prd SKILL.md SHALL include examples showing GUID format for Feature ID, User Story IDs (US-[FEATURE-GUID]-01), Functional Requirement IDs (FR-[FEATURE-GUID]-01), Acceptance Criteria IDs (AC-[FEATURE-GUID]-01-01), and Test Case IDs (TC-[FEATURE-GUID]-01-01)
3. THE cc-gen-prd-lite SKILL.md SHALL include examples showing GUID format for Feature ID, User Story IDs (US-[FEATURE-GUID]-01), Functional Requirement IDs (FR-[FEATURE-GUID]-01), Acceptance Criteria IDs (AC-[FEATURE-GUID]-01-01), and Test Case IDs (TC-[FEATURE-GUID]-01-01)
4. THE cc-gen-tech-spec SKILL.md SHALL include examples showing GUID-based user story references in Design Details sections
5. THE cc-gen-prd-task SKILL.md SHALL include examples showing both GUID and sequential ID formats
6. THE cc-gen-prd-task SKILL.md SHALL document the conversion rules for GUID-based IDs
7. WHEN examples show Business Objective IDs, THE Examples SHALL use the format "OBJ-[FEATURE-GUID]-01"
8. WHEN examples show User Story headings, THE Examples SHALL use the format "### US-[FEATURE-GUID]-01: [Title]"
9. WHEN examples show Functional Requirements, THE Examples SHALL use the format "FR-[FEATURE-GUID]-01: [Requirement] (Supports US-[FEATURE-GUID]-XX)"
10. WHEN examples show Acceptance Criteria, THE Examples SHALL use the format "AC-[FEATURE-GUID]-01-01", "AC-[FEATURE-GUID]-01-02"
11. WHEN examples show Test Cases, THE Examples SHALL use the format "TC-[FEATURE-GUID]-01-01" with "Validates: US-[FEATURE-GUID]-01, AC-[FEATURE-GUID]-01-01"

### Requirement 10: Update cc-gen-brd-prd Template Files

**User Story:** As a skill user, I want the template files to demonstrate GUID-based ID formats, so that I can see concrete examples of the new ID structure.

#### Acceptance Criteria

1. THE BRD_PRD_Template.md file SHALL be updated to show GUID-based examples for all ID types
2. THE BRD_Template.md file SHALL be updated to show GUID-based examples for Business Objective IDs
3. THE PRD_Template.md file SHALL be updated to show GUID-based examples for User Story IDs, Functional Requirement IDs, and Acceptance Criteria IDs
4. THE Complete_User_Story_Template_with_Gherkin.md file SHALL be updated to show GUID-based User Story ID and Acceptance Criteria ID examples
5. ALL template files SHALL include a note explaining that GUIDs use UUID v4 format
6. THE template files SHALL show the hierarchical relationship: Feature GUID → US-[GUID]-01 → AC-[GUID]-01-01
7. THE template files SHALL maintain backward compatibility notes indicating sequential IDs (US-001) are still valid for existing documents

### Requirement 11: Maintain Backward Compatibility

**User Story:** As a user with existing PRDs, I want the skills to continue working with my old sequential IDs, so that I don't have to migrate all my existing documents.

#### Acceptance Criteria

1. WHEN cc-gen-prd-task converts a PRD with sequential IDs (001, 002, US-001, FR-1, OBJ-01), THE Skill SHALL process it successfully
2. WHEN cc-gen-prd-task encounters a PRD with date-based IDs (2026-02-17), THE Skill SHALL process it successfully
3. THE Skills SHALL NOT automatically convert existing sequential IDs to GUIDs
4. THE Skills SHALL generate GUIDs only for newly created PRDs
5. THE prd.json format SHALL accept both GUID and sequential formats in the id and featureId fields

### Requirement 12: Provide GUID Generation Guidance

**User Story:** As a skill maintainer, I want clear guidance on when and how to generate GUIDs, so that the implementation is consistent across all skills.

#### Acceptance Criteria

1. THE SKILL.md files SHALL document when GUIDs are generated (at PRD creation time)
2. THE SKILL.md files SHALL specify that GUIDs use UUID v4 format
3. THE SKILL.md files SHALL explain that GUIDs provide better traceability with external systems
4. THE cc-gen-prd-task SKILL.md SHALL document how to handle both GUID and sequential ID formats
5. THE SKILL.md files SHALL include a note that existing PRDs with sequential IDs remain valid

## Non-Goals (Out of Scope)

- Automatic migration of existing PRDs from sequential IDs to GUIDs
- Custom GUID formats or prefixes beyond standard UUID v4
- Integration with external GUID generation services
- GUID validation or uniqueness checking across the entire repository
- Support for UUID versions other than v4 (v1, v3, v5)

## Success Metrics

- All five PRD-generating and processing skills successfully handle GUIDs
- cc-gen-tech-spec correctly preserves GUID-based IDs from PRDs in technical specifications
- cc-gen-prd-task successfully converts PRDs with both GUID and sequential ID formats
- Existing PRDs with sequential IDs continue to work without modification
- Documentation and examples clearly demonstrate GUID usage across the entire workflow
- Zero breaking changes for users with existing PRDs

## Open Questions

None at this time. The requirements are clear and implementation is straightforward.
